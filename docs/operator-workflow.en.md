# Operator Workflow Proof

[中文](operator-workflow.zh-CN.md) | [Documentation](README.en.md) | [Release Operations](release-operations.en.md) | [Current Design](current-design.md)

A KnowMesh operator should not edit databases or move artifacts by hand. Operators should use the Web Console and scoped HTTP APIs to complete the knowledge-asset lifecycle: inspect source folders, confirm execution plans, recover or stop jobs, review quality issues, trigger targeted reruns, publish versions, compare diffs, preview rollback risk, and confirm rollback.

`0.8.0 Operator Workflow Proof` proves that this path is reviewable by real maintainers, not only implied by internal implementation details.

## Operator Path

1. Create a non-sample `general-docs` knowledge base.
2. Select local source and workspace folders.
3. Run folder precheck, scan preview, source manifest, and execution plan preview.
4. Exclude and restore documents, then rescan changed / missing / restored source deltas.
5. Confirm a local job and inspect checkpoints, progress, logs, and the current task summary.
6. Verify pause, resume, and stop semantics, then recover the latest job after service restart.
7. Use evidence search, query feedback, and quality issues to enter maintenance review.
8. Generate a safe rerun scope, confirm targeted rerun, and record review resolution.
9. Publish a catalog-backed version, then inspect package preview, version list, diff, and rollback preview.
10. Confirm rollback, then verify active/previous version state and cross-KB isolation.

The authority for this path remains `workspace.sqlite` plus each knowledge base's `catalog.sqlite`. JSON/JSONL may remain only as export, audit, sidecar, checkpoint, credential, schema, or template boundaries; it must not return as mutable primary state.

## Acceptance Commands

```bash
npm run smoke:operator-workflow
node ./scripts/release-gate.mjs --stage operator-workflow --evidence ./release-evidence.json
```

`smoke:operator-workflow` starts a temporary local service and temporary folders, creates a non-sample operator KB, exercises source intake, execution recovery, maintenance targeted rerun, version rollback, desktop/narrow DOM contracts, and privacy audit, then removes temporary state.

Release evidence can be generated by `scripts/generate-release-evidence.mjs` or supplied explicitly:

```bash
node ./scripts/generate-release-evidence.mjs \
  --operator-workflow \
  --source-intake-proof pass \
  --execution-recovery-proof pass \
  --maintenance-targeted-rerun-proof pass \
  --version-rollback-proof pass \
  --operator-browser-workflow pass \
  --operator-privacy-audit-proof pass \
  --operator-package-asset-review pass
```

## Disallowed Shortcuts

- Operators must not directly read or mutate `catalog.sqlite` to perform maintenance actions.
- Browser storage must not store selected KB, setup draft, job summary, or release decisions as truth.
- Diagnostic exports must not leak local paths, credentials, raw provider payloads, private document text, or internal artifacts.
- Replaced JSON-first mutable state flows must not be preserved as compatibility shims.
- A version that fails rollback preview must not become active.

## Contributor Entry Points

Good operator-facing contribution tasks include:

- Improve source delta explanations and document-maintenance prompts.
- Add more auditable previews for targeted rerun scopes.
- Improve version diff readability and rollback-risk copy.
- Expand Web Console operator DOM-contract coverage without using browser storage as truth.
- Extend `operatorPackageAssetReview` so release-asset review better matches real publication workflows.
