# First-Run Usability Proof

[中文](first-run-usability.zh-CN.md) | [Documentation](README.en.md) | [Operator Workflow Proof](operator-workflow.en.md) | [Current Design](current-design.md)

`0.9.0 First-Run Usability Proof` compresses the proven operator workflow into the path a non-maintainer should be able to complete on their first KnowMesh run: start from empty local state, create a knowledge base, choose source folders, confirm a build, recover a job, ask the first question, and turn feedback into a clear maintenance next action.

This is not a second product blueprint. State authority remains `workspace.sqlite` plus each knowledge base's `catalog.sqlite`; browser storage is limited to visual preferences, and JSON/JSONL may appear only as export, audit, sidecar, checkpoint, credential, schema, or template boundaries.

## First-Run Path

1. Start the local service with an empty user data root; home and knowledge-base management must not create an implicit knowledge base.
2. Expose create-KB and public-sample actions, plus runtime, provider, and integration readiness without credentials.
3. Create two `general-docs` knowledge bases, switch the current selection, and verify scoped overview, setup, and diagnostics follow the selected KB.
4. Choose local source and workspace folders, save setup draft and retrieval strategy, then recover them after service restart.
5. Block missing source folders with precheck, then explain the next step through scan preview and plan preview.
6. Confirm a local build and show progress, pause/resume, restart recovery, and completion.
7. Reach Query Runtime and return either a cited answer or explicit no-answer/refusal; weak answers must not count as success.
8. Store feedback in maintenance review and expose a safe rerun scope without leaking internal tables or file paths.

## Acceptance Commands

```bash
npm run smoke:first-run-usability
node ./scripts/release-gate.mjs --stage first-run-usability --evidence ./release-evidence.json
```

`smoke:first-run-usability` starts a temporary local service with empty local state, creates a non-sample `general-docs` knowledge base, exercises launch, create/select, guided setup, build recovery, first question, maintenance next action, desktop/narrow DOM contracts, and privacy cleanup, then removes temporary state.

Release evidence can be generated by `scripts/generate-release-evidence.mjs` or supplied explicitly:

```bash
node ./scripts/generate-release-evidence.mjs \
  --first-run-usability \
  --first-run-launch-proof pass \
  --guided-setup-proof pass \
  --build-recovery-proof pass \
  --first-question-proof pass \
  --maintenance-next-action-proof pass \
  --first-run-browser-workflow pass \
  --first-run-package-asset-review pass
```

## Disallowed Shortcuts

- Do not create a hidden default knowledge base at startup.
- Browser storage must not store selected KB, setup draft, job summary, or first-run progress as truth.
- The `general-docs` first-run path must not inherit K12 required fields.
- Weak answers must not appear as successful answers when evidence is insufficient.
- Diagnostic exports must not leak credentials, private content, local paths, raw provider payloads, or internal artifacts.

## Contributor Entry Points

Good first-run contribution tasks include:

- Improve empty-state, folder-precheck, and scan-preview copy.
- Add reviewable fixtures and next-step guidance for common build failures.
- Improve no-answer/refusal explanations and feedback entry points.
- Expand desktop/narrow DOM contracts for `firstRunBrowserWorkflow`.
- Strengthen `firstRunPackageAssetReview` so release-asset review covers more real publication risks.
